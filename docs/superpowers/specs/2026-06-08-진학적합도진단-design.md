## 1. 배경 & 가능성 진단 (Feasibility)

### 1.1 "나이스"는 두 개다 — 데이터 진입의 핵심 제약

| 구분 | 개방 포털 OpenAPI (open.neis.go.kr) | 나이스 학부모서비스 (parents.neis.go.kr) |
|---|---|---|
| 성격 | 학교 단위 **공개** 데이터 API | 학생 **개인** 데이터 열람 포털 |
| 인증 | 인증키 신청 → 누구나 호출 | 본인인증(공동/간편) 로그인 필수 |
| API 제공 | 있음 (REST, JSON/XML) | **없음** (화면 열람 + PDF 발급만) |
| 생기부/성적/출결 | ❌ 제공 안 함 | ✅ 열람 가능하나 PDF로만, API 없음 |

**결정적 함의:** 학생의 생활기록부·성적·출결·세특·표준점수는 **어떤 공식 API로도 받을 수 없다.**
학부모서비스 자동 스크래핑은 약관·개인정보보호법 위반 소지가 크다.

→ 유일한 현실적 데이터 진입: **학부모/학생이 학부모서비스에서 생기부 PDF 발급 → 앱에 업로드 → AI 파싱.**
학원이라는 정당한 관계 + 본인 동의가 있어 합법적.

### 1.2 OpenAPI로 받을 수 있는 것 (학교 단위, 보조 메타데이터)

학교기본정보 · 학교계열정보 · 학교학과정보 · 학사일정 · 초/중/고 시간표 · 급식식단 · 학원교습소정보 등.
개인 진로 분석의 **메인 엔진이 아니라 보조**. Phase 1에선 생략 가능(루브릭은 Claude 지식만으로 완결).

### 1.3 자기소개서는 폐지됐다

2024학년도 대입부터 학생부종합전형 **자기소개서 전면 폐지**. 현재 학종 = 생기부 + 면접.
수상경력·자율동아리·독서활동도 대입 미반영. 내신·세특·창체의 영향력이 핵심.

→ "좋은 자소서 참고 가이딩"은 현행 대입 본질과 불일치. 대신 **세특/창체 코칭, 면접 대비**로 에너지 전환.
(자소서가 유효한 영역: 특기자전형·특목고/자사고 입시·취업 자소서.)

## 2. 목표 & 시퀀싱

**1차 목표:** 진학 적합도 진단 — 생기부를 분석해 "목표 학과/계열 적합도 + 강점/공백"을 리포트.

**비교 기준 3종 (목적지) 과 시퀀싱:**

| | 데이터 의존성 | 난이도 | 설득력 | 법적 리스크 |
|---|---|---|---|---|
| ① 인재상 루브릭 (정성, AI 생성) | 없음 | 낮음 | 중 | 낮음 |
| ② 합격 사례 DB (설득) | 높음 (직접 축적) | 중 | 높음 | 중 |
| ③ 공식 커트라인 (정량) | 외부 수급 | 높음 | 중 | 낮음 |

**시퀀싱 결정 — ① → ② → ③ (데이터 현실 기반):**
사례 데이터가 "파편적 기록(결과 라벨)만" 존재하고 동의·비식별화 미정리.
→ ②를 지금 만들 수 없음(코퍼스 본체 부재 + 법적 토대 부재).
→ **① 루브릭 진단을 Phase 1로**, 그것이 굴러가며 ②의 코퍼스를 *축적*하는 플라이휠 구조.

```
Phase 1: ① 루브릭 진단
   생기부 PDF → 구조화 파싱 → 학과 루브릭 채점 → 리포트
        ├─ 매 사용마다 "구조화 생기부 1건" 축적  ← ②의 코퍼스
        ├─ 온보딩에 동의 플로우 내장             ← ②의 법적 토대
        ▼ (입시 시즌 결과 라벨 태깅)
Phase 2: ② 사례DB 비교 (저절로 성숙)
Phase 3: ③ 커트라인 정량 보정 + NEIS 학사일정
```

파싱·루브릭·리포트 엔진은 ②③에 100% 재사용 — 버리는 것 없음.

## 3. 아키텍처 (기존 5-layer 준수)

저장·UI는 기존 Notion 스택에 얹는다. 주간 리포트의 패턴 D(HTML 드래프트 → 원장 승인 → Notion)를 복제.

### 3.1 Layer 3 — 엔진 4종 (`intelligence/`)

| 모듈 | 입력 → 출력 | 비고 |
|---|---|---|
| `saenggibu_parser.py` | 생기부 PDF → 구조화 JSON (교과세특·창체·행특·출결·성적추세, 학년별) | 전체의 토대. ②③ 재사용 |
| `rubric_builder.py` | 목표 학과/계열 → 인재상·핵심역량 채점 루브릭(JSON) | 학과당 1회 캐시 |
| `fit_scorer.py` | 구조화 생기부 + 루브릭 → 역량별 점수·근거·공백 | **근거 인용 강제** (생기부 출처 항목 명시) |
| `career_report.py` | 채점 결과 → 학부모/학생 리포트 (강점·공백·다음 액션) | `weekly_report.py` 패턴 복제 |

프롬프트 본문은 `intelligence/prompts/*.md` 에 분리 (학원 톤·정책 분리 원칙).

**설계 원칙:**
- 모든 Claude 출력은 pydantic 스키마 검증 후 저장. 파싱 실패 항목은 `null` + 플래그(원장 검수).
- 루브릭은 "정답"이 아니라 *가설*. 리포트에 "공식 입시 기준 아닌 인재상 기반 참고 진단" 명시 — 과신 방지.

### 3.2 Layer 2 — 데이터 모델 (`storage/notion_repo.py`, `PROP_*` 패턴)

기존 5개 DB → 7개. 새 DB 2개:

**DB 6: 진로 프로필** (학생당 1개, 운영 채널)

| 컬럼 | 타입 | 비고 |
|---|---|---|
| 학생 | Relation → 학생 Master | |
| 목표 대학/학과 | Text | 복수 가능 |
| 생기부 원본 | Files | 업로드 PDF (동의 시에만) |
| 구조화 생기부 | 페이지 children | 파싱 JSON 블록 |
| 최신 적합도 점수 | Number | |
| 동의 상태 | Select | 미동의 / 내부분석 / 코퍼스활용 |
| 동의일시 | Date | |

**DB 7: 진학 코퍼스** (②의 씨앗 — 비식별 구조화 생기부)

| 컬럼 | 타입 | 비고 |
|---|---|---|
| 코퍼스 ID | Title | 익명 키 (학생 식별 불가) |
| 구조화 생기부 | children | **비식별** 버전만 |
| 계열/학과 태그 | Multi-select | |
| 결과 라벨 | Select | 미정 / 합격 / 불합격 (입시 시즌 태깅) |
| 동의 근거 | Text | 동의 레코드 참조 |

DB7은 DB6과 **물리적으로 분리**. 식별자 제거 후에만 적재. ②의 법적·데이터 토대를 처음부터 깐다.

### 3.3 Layer 4 — 파이프라인 (`pipelines/career.py`)

```
[온보딩] 동의 → 동의 상태 기록 (DB6)
     │ (미동의면 업로드 자체 차단)
     ▼
[업로드] 생기부 PDF → pipelines/career.py
     ▼
① saenggibu_parser → 구조화 JSON → pydantic 검증(실패=플래그)
     ▼
② rubric_builder   → 루브릭 (학과별 캐시 우선)
     ▼
③ fit_scorer       → 점수·근거·공백
     ▼
④ career_report    → HTML 드래프트 (reports_out/career/<id>.html)
     ▼
[승인 게이트] 원장 검수 → 승인 시에만:
     ├─ DB6 진로 프로필 아카이브
     ├─ '코퍼스활용' 동의 시 → 비식별화 → DB7 적재
     └─ notify/ 로 학부모 상담 문구 발송 (Layer 5 재사용)
```

**동의 3단계 = 코드 차원 강제 게이트:**
- `미동의` → 업로드·분석 전면 차단
- `내부분석` → DB6 분석만, DB7 적재 안 함
- `코퍼스활용` → 비식별 버전이 DB7로

사람이 깜빡해도 미동의면 시스템이 막는다.

### 3.4 NEIS OpenAPI 연동 (Phase 1 보조, 생략 가능)

- `intelligence/neis_reference.py` (read-only 참조, ClassIn과 무관해 `classin/` 에 넣지 않음)
- 용도: 학교학과정보·계열정보로 루브릭 메타 보강, 학사일정 리마인드(나중)
- MVP에선 생략 — 루브릭은 Claude 지식만으로 완결

## 4. 엣지 케이스 & 테스트

| 상황 | 처리 |
|---|---|
| 스캔 PDF(이미지) | OCR 경로 분기, 신뢰도 낮으면 검수 플래그 |
| 파싱 일부 실패 | 항목 `null` + 플래그, 점수에서 제외(0점 아님) |
| 목표 학과 미정 | 계열 단위 루브릭 폴백 |
| 동의 철회 | DB7 비식별 레코드 삭제 경로 제공 |
| Claude 환각(없는 근거 인용) | fit_scorer 출력 인용을 원본 생기부와 대조 검증 |

**테스트 (외부 mock 없이 도는 원칙):**
- 합성 생기부 픽스처 1~2건 → 파서 스키마 골든 테스트
- 루브릭/채점은 결정적 입력의 구조 검증(점수 범위·근거 필드 존재)
- 동의 게이트는 순수 로직 단위 테스트(미동의→차단)
