# ClassIn 서드파티 자동화 컨설팅 프로젝트
## 운영 지침

---

## 1. API 활용 지침

### 1.1 CED API 운영 원칙
- API 호출로 생성된 모든 ID값(CourseID 등)은 **반드시 Notion DB에 저장**. 수정·삭제 시 이 ID가 키값이 됨.
- 계정 생성 시 비밀번호 직접 설정 (SMS 인증 생략됨) → 초기 비밀번호 관리 정책 학원과 합의 필요.
- **ClassIn 대시보드와 API를 동시에 열어두고 조작 금지**. 데이터 충돌 발생.
- 학원에 반드시 고지: "API로 등록된 데이터는 대시보드에서 수동 수정 시 충돌 위험"

### 1.2 Webhook 운영 원칙
- 1기관당 Webhook 엔드포인트 **1개만 허용**. 엔드포인트 변경 시 ClassIn 재등록 필요.
- **Real-time 데이터 사용 금지** (동시 접속 많을 시 서버 다운 및 3일 이상 지연 위험).
- **After-Class 데이터만 사용** (수업 종료 후 20분 이내 안정적 전송).
- Webhook 등록 시 담당자 이메일 반드시 등록 (에러 즉시 알림 수신).
- 수신 실패에 대비한 재전송 정책 ClassIn에 확인 필요.

### 1.3 데이터 처리 원칙
- ClassIn이 제공하는 것: **.mp4 URL만** (녹화본). 영구 보관 필요 시 자체 스토리지 다운로드 필수.
- LMS 성적 조회 API 없음 → 성적 데이터는 오프라인 입력 또는 Claude OCR로 처리.
- After-Class 학생 데이터 항목: 손들기 횟수, 트로피 수, Poll 응답, 카메라 켜둔 시간, 출석/지각 여부.

## 2. 개발 지침

### 2.1 아키텍처 원칙
- **Layer 1 (API 래퍼)는 반드시 분리** 설계. ClassIn API 변경 시 이 레이어만 수정.
- **비즈니스 로직(Layer 4)과 출력 계층(Layer 5) 분리**. 로컬 → 웹 전환 시 Layer 5만 교체.
- 학원별 커스터마이징은 **코드 수정 없이 config.yaml 변경**으로 가능하도록 설계.

### 2.2 로컬 설치 패키지 구성
```
classin-toolkit/
├── setup.exe (or .sh)     ← 설치 마법사
├── config.yaml            ← API 키, Notion 토큰 등 세팅
├── webhook_receiver.py    ← 로컬 수신기 (Cloudflare Tunnel 포함)
├── reports/               ← 리포트 생성 로직 (Claude 프롬프트)
└── README.md              ← 학원 운영 가이드
```

### 2.3 Webhook 수신 환경
- **개발·테스트**: ngrok 또는 Cloudflare Tunnel로 로컬 임시 노출
- **운영 (권장)**: Supabase Edge Function (무료 티어) → DB 적재 → 로컬 앱이 pull
- 학원 PC에 설치 시: Cloudflare Tunnel을 패키지에 포함, 절전모드 해제 가이드 동봉
- 완전 무결성 필요 시: 라즈베리파이 상시 가동 옵션 (5~10만원) 제공

### 2.4 Claude 활용 원칙
- Claude는 **"분석·리포트·문구 생성·OCR"** 역할에만 집중. 데이터 수집·저장은 별도 처리.
- 학생별로 **다른 리포트**가 나오도록 프롬프트 설계. 일괄 복붙 형태는 학부모가 즉시 알아챔.
- 한국 학원 문화에 맞는 톤앤매너 프롬프트 별도 관리 (카톡 문구 ≠ 공식 보고서 문체).
- 프롬프트 템플릿은 학원 티어별로 분리 관리.

### 2.5 MVP 개발 시 버릴 것들
| 항목 | 이유 |
|------|------|
| 예쁜 웹 UI | Notion으로 대체, 초기엔 불필요 |
| 멀티테넌시 구조 | 파일럿 1곳이면 불필요 |
| 인증·로그인 시스템 | 로컬 설치 = OS 로그인이 인증 |
| 완벽한 에러 처리 | 데모 단계에선 과잉 투자 |
| 자동 업데이트 기능 | 실제 판매 시작 후 추가 |
| 카톡 알림톡 실제 연동 | MVP 단계는 예시 문구로 충분 |

## 3. 영업·컨설팅 지침

### 3.1 데모 시나리오 원칙
- MVP 완성 기준: **3~5분 데모 영상 촬영 가능 상태**
- 데모 학원 설정: 가상 "○○수학학원" + 5명 페르소나 (성실형/지각형/참여도 하락형 등)
- 각 페르소나별 리포트가 **반드시 다르게** 출력되도록 구성
- 원장님 반응 목표: "이거 얼마면 살게요"

### 3.2 제안서 구조
1. 이런 거 귀찮으시죠? (문제 공감)
2. 이걸 보여드립니다 (데모 영상 링크)
3. 어떻게 사용하는지 (기존 ClassIn 방식 유지)
4. 가격 (초기 설치비 + 월 유지비)
5. 다음 단계 (14일 무료 체험 제안)

### 3.3 고객 교육 포인트 (필수 고지)
- API와 ClassIn 대시보드 동시 조작 금지
- 학원 PC 상시 가동 필요 (절전 해제)
- 데이터 주권은 학원에 있음 (학원이 처리자)
- API 키는 학원이 직접 보관·관리

### 3.4 계약 구조
- 학원이 **개인정보 처리자**, MOON은 **수탁자** 구조로 계약
- SLA(서비스 가용성), 데이터 삭제 정책, 업데이트 범위 명시
- ClassIn API 변경 시 긴급 업데이트 조건 별도 명시

## 4. 카톡 알림 운영 지침

### 4.1 서비스 선택
- **알리고** 또는 **솔라피** 릴레이 서비스 사용 (사업자등록 필요)
- 카카오 알림톡 템플릿 심사 기간: 최소 2~3주 → MVP 단계에서는 예시 문구만 제시
- 실제 발송은 Standard 티어부터 포함

### 4.2 알림 유형 및 트리거
| 알림 종류 | 트리거 | 수신자 |
|-----------|--------|--------|
| 숙제 미제출 알림 | After-Class 데이터 수신 후 | 학생·학부모 |
| 지각/결석 알림 | 출석 데이터 이상 감지 | 학부모 |
| 주간 리포트 발송 | 매주 금요일 자동 | 학부모 |
| 장기 결석 경고 | 3주 연속 결석 패턴 | 원장·학부모 |
| 상담 일정 확인 | 상담 전날 자동 | 학부모 |

### 4.3 문구 작성 원칙
- Claude가 생성하되, 학원명·교사명·학생명 치환 변수 활용
- 너무 딱딱하지 않게 (학원 카톡 = 학부모와의 관계 유지 도구)
- 미제출 알림은 부드럽게, 반복 미제출은 단계적으로 톤 강화

## 5. Notion DB 운영 지침

### 5.1 기본 DB 구조
- **학생 Master DB**: 이름, 반, 학부모 연락처, ClassIn ID
- **수업 기록 DB**: 날짜, 참여 지표, 출석 여부 (ClassIn After-Class 데이터)
- **성적 DB**: 모의고사, 내신, 자체 시험 (오프라인 입력)
- **리포트 DB**: 주간/월간 리포트 보관, 발송 여부 추적

### 5.2 원칙
- ClassIn이 반환한 **ID값은 반드시 학생 Master DB에 저장**
- 수업 기록은 ClassIn에서 받은 그대로 + 오프라인 데이터 merge (학생 ID 기준)
- 리포트는 Notion 페이지로 보관, 학부모 발송용 문구는 별도 컬럼

### 5.3 접근 권한
- 원장: 전체 뷰 접근
- 교사: 본인 담당 학생만 필터된 뷰
- MOON(컨설턴트): 세팅 및 유지보수용 접근 (계약 기간 한정)
