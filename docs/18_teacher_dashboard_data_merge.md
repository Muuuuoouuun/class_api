# 선생님 상황판 + 로컬 데이터 병합

이 문서는 선생님이 실제로 매일 쓰는 화면의 기준을 정의한다. 선생님은 바쁘고, 반복 입력을 싫어하고,
어려운 설정을 오래 붙잡고 있을 시간이 없다. 그래서 상황판은 "데이터 관리 도구"가 아니라
"오늘 처리할 학생을 바로 알려주는 작업판"이어야 한다.

## 1. 핵심 질문

상황판은 아래 질문에 10초 안에 답해야 한다.

- 오늘 숙제 안 한 학생은 누구인가?
- 누구에게 이미 연락이 갔는가?
- 아직 연락이 안 간 학생은 누구인가?
- 연락처가 없거나 발송 실패한 학생은 누구인가?
- 반복 미제출, 결석, 성적 하락처럼 상담이 필요한 학생은 누구인가?
- 이번 주 보고서에 넣을 만한 특이사항은 무엇인가?

## 2. 선생님용 첫 화면

첫 화면은 기능 메뉴가 아니라 우선순위 큐다.

```text
[오늘의 숙제/상담 관리]

숙제 미제출 6명 | 연락 필요 2명 | 실패 1명 | 연락처 없음 1명 | 상담 후보 3명

[전체] [연락 필요] [다시 확인] [연락처 없음] [완료]

김지각
수학 숙제 미제출 · 2회째 · 보호자에게 아직 안 보냄
[보호자에게 보내기]

이하락
참여도 하락 · 지난 연락 실패 · 이번 주 보고서에 상담 권장
[다시 보내기]

최결석
3주 연속 결석 · 보호자 연락처 없음
[연락처 입력]
```

원칙:

- 한 학생에게 가장 필요한 버튼 하나를 크게 보여준다.
- `pending`, `dry_run`, `failed` 같은 내부 용어는 UI에 직접 노출하지 않는다.
- 테이블보다 학생 중심 행을 우선한다.
- "누구에게 언제 보냈는지"는 별도 이력 화면이 아니라 학생 행 안에 표시한다.

## 3. 보고서와 외부 데이터 병합

ClassIn 데이터만으로는 학원 운영 맥락이 부족하다. 주간 보고서와 상황판은 아래 데이터를 함께 읽어야 한다.

| 데이터 | 예시 | 보관 위치 | 병합 키 |
|---|---|---|---|
| ClassIn 수업 기록 | 출석, 숙제 제출, 손들기, 트로피, 카메라 | Notion 수업 기록 DB | `student_classin_id`, 수업일 |
| 주간/일일 보고서 | HTML 드래프트, 승인된 리포트, 학부모 문구 | `reports_out/` 또는 Notion 리포트 DB | 학생 ID, 리포트 기간 |
| 로컬 CSV/XLSX | 오프라인 출결, 보강, 보충수업, 숙제 검사표 | 학원 PC 공유 폴더 | 학생명 + 반 + 날짜 |
| 오프라인 성적 | 내신, 모의고사, 자체 시험, 단원평가 | 로컬 파일 또는 Notion 확장 DB | 학생명 + 시험일 + 과목 |
| 상담/메모 | 전화 상담, 학부모 요청, 선생님 코멘트 | Notion 메모 DB 또는 로컬 메모 파일 | 학생 ID, 작성일 |
| 공유 자료 | 선생님이 전달한 PDF, 이미지, 스캔본 | `local_data/inbox/` 같은 로컬 폴더 | 파일명, 업로드일, 수동 매칭 |

## 4. 병합 방식

초기 버전은 단순하고 안전해야 한다.

```text
local_data/inbox/
  ├─ attendance/*.csv
  ├─ scores/*.xlsx
  ├─ memos/*.md
  └─ attachments/*

        ↓

local_data/normalized/
  ├─ attendance.jsonl
  ├─ scores.jsonl
  ├─ memos.jsonl
  └─ attachments.jsonl

        ↓

학생별 timeline/context 생성
        ↓

상황판의 상담 후보, 주간 보고서, AI 질문 응답에 반영
```

현재 구현은 정규화 파일을 별도로 쓰지 않고, `src/classin_toolkit/pipelines/data_merge.py`에서
읽기 전용으로 메모리 context를 만든다. 지원하는 입력 규칙은 아래와 같다.

- `attendance/*.csv`: `student_classin_id` 또는 `classin_id`, `student_name`, `class_name`, `date`, `attendance` 컬럼을 우선 사용한다.
- `scores/*.xlsx`: `student_classin_id` 또는 `classin_id`, `student_name`, `class_name`, `subject`, `score`, `date` 컬럼을 우선 사용한다.
- `memos/*.md`: 상단 12줄 안의 `student_classin_id`, `student_name`, `class_name`, `date` 메타데이터를 읽는다.
- `attachments/*`: 파일명에 ClassIn ID가 있거나 같은 이름의 JSON 보조 파일에 `student_classin_id`가 있으면 자동 연결한다.
- 자동 연결이 되지 않은 항목은 `needs_review_items`로 남기고, 원본 파일은 수정하지 않는다.

규칙:

- 원본 파일은 삭제하거나 덮어쓰지 않는다.
- 자동 매칭이 애매하면 "확인 필요" 상태로 남긴다.
- 학생명만으로 강제 병합하지 않는다. 가능하면 ClassIn ID, 전화번호, 반, 날짜를 함께 본다.
- 보고서 생성 시에는 병합된 외부 데이터의 출처를 짧게 남긴다.
- 보호자 발송 전에는 외부 데이터 기반 민감 문구를 선생님이 확인할 수 있어야 한다.

## 5. UI에서 보이는 방식

선생님에게는 파일 병합 과정이 복잡하게 보이면 안 된다. 화면에는 아래 정도만 보여준다.

- `새로 들어온 공유 자료 3개`
- `자동 연결 완료 2개`
- `학생 확인 필요 1개`
- 학생 행 안의 작은 표시: `오프라인 시험 1건`, `상담 메모 2건`, `보강 출석 있음`

학생 상세 또는 보고서 미리보기에서는 이렇게 보인다.

```text
[학생 맥락]
- ClassIn: 최근 3회 중 숙제 2회 미제출
- 오프라인: 4/22 단원평가 68점
- 상담: 4/24 학부모가 보강 가능 시간 문의
- 보고서 반영: 학습 루틴 회복 + 보강 제안 톤 권장
```

## 6. 구현 우선순위

1. 상황판의 학생 행에 `report_context` 필드를 추가한다.
2. `reports_out/weekly`의 드래프트/승인 상태를 학생별로 읽어 상황판에 붙인다.
3. `local_data/inbox` 폴더 규칙을 만들고 CSV/XLSX 샘플을 1개씩 지원한다.
4. 학생명 + 반 + 날짜 기반의 보수적 매칭을 만든다.
5. 매칭 실패/중복 후보를 상황판의 `확인 필요` 큐에 올린다.
6. 주간 리포트 프롬프트에 병합된 외부 데이터를 근거로 넣되, 출처를 함께 제공한다.

## 7. AI 작업 지시 기준

AI에게 이 기능을 시킬 때는 이렇게 지시한다.

```text
선생님 상황판에 보고서와 로컬/오프라인 공유 데이터를 병합하는 기능을 추가하세요.
선생님 화면은 복잡한 데이터 관리 UI가 아니라 오늘 처리할 학생 큐여야 합니다.

우선 reports_out/weekly의 리포트 상태와 local_data/inbox의 CSV/XLSX 샘플을 읽어
학생별 context로 정규화하세요. 원본 파일은 절대 수정하지 말고,
자동 매칭이 애매한 항목은 확인 필요 큐로 올리세요.

병합 결과는 미제출 상황판, 주간 보고서 생성, AI 질문 응답에서 재사용되게 하세요.
```
